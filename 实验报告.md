# 课程资料智能摘要与测验生成系统实验报告

## 1 绪论

### 1.1 研究背景

随着人工智能技术的迅速发展，大语言模型（Large Language Model，LLM）在自然语言处理领域展现出强大的文本理解与生成能力。以GPT系列、通义千问、DeepSeek等为代表的大模型，通过海量数据的预训练，能够在翻译、摘要、问答、代码生成等多种任务上取得优异表现，且无需针对每个任务单独训练。

在教育领域，课程资料的整理、摘要生成、知识点提炼和测验出题是教师和学生的高频需求。传统方式依赖人工操作，效率低且质量参差不齐。基于大语言模型的自动化处理方案能够有效减轻教师负担、提升学生学习效率，具有重要的应用价值。

然而，在实际应用中存在一个核心问题：对于同一个任务，是让模型一次性完成效果更好，还是将任务拆解为多个子步骤分步完成？这涉及到推理策略的选择——端到端推理与分步推理在输出质量、计算成本和格式规范性上存在差异。本研究围绕这一问题展开实验分析。

### 1.2 研究目的与意义

本研究旨在构建一个完整的课程资料智能处理系统，实现对课程PDF、TXT等格式文本的自动摘要、知识点提纲生成和选择题出题功能。在此基础上，设计对比实验，探究**单轮直接生成**与**多轮分步迭代**两种推理方案在摘要生成和选择题生成两个任务上的效果差异。

研究意义体现在以下方面：一是为教育场景下的自动化课程资料处理提供可用的系统方案；二是通过对比实验分析不同推理策略的优劣，为大语言模型在结构化文本生成任务中的应用提供参考；三是探索在有限API调用条件下，如何通过策略设计提升输出质量。

### 1.3 国内外研究现状

在课程摘要与测验生成方面，国内外已有一定研究基础。早期研究主要基于抽取式方法，即从原文中选取关键句组成摘要（Luhn，1958），此后序列到序列模型的引入（Rush等，2015）使生成式摘要成为主流。BERT的提出（Devlin等，2018）推动了预训练语言模型在下游NLP任务中的广泛应用。GPT系列的出现（Radford等，2018；Brown等，2020）则使生成式AI在教育领域的应用成为可能。

在推理策略研究方面，思维链（Chain-of-Thought，CoT）推理被证明能显著提升模型在复杂推理任务上的表现（Wei等，2022）。Brown等（2020）提出In-Context Learning，展示了通过示例引导模型解决新任务的能力。这些研究表明，提示策略的设计对大模型输出质量有重要影响。但现有研究多集中在数学推理和常识问答领域，对课程摘要和知识点出题这类结构化生成任务的研究相对较少。

在国内，通义千问、文心一言、DeepSeek等国产大模型的发布，为中文教育场景的智能化处理提供了新的技术路径。本研究基于通义千问（qwen-plus）模型，结合实际课程资料，系统对比不同推理策略的效果。

### 1.4 研究内容

本研究围绕以下内容展开：

（1）设计并实现课程资料智能处理系统，包含资料导入、摘要生成、知识点提纲生成、选择题生成和结果导出五个功能模块。

（2）针对摘要生成和选择题生成两个任务，分别设计两种推理方案：方案一为单轮直接生成（一次API调用完成全部任务），方案二为多轮分步迭代（将任务拆解为2-3个子步骤，依次调用API）。

（3）在同一课程文本输入下，从输出字数、耗时、API调用次数等客观指标和内容质量等人为主观评分两个维度进行对比分析。

（4）基于Streamlit框架搭建Web交互界面，实现文件上传、功能操作、结果展示与导出的一体化操作流程。

（5）进阶功能部分：引入RAG（检索增强生成）架构，将系统从单章处理扩展为支持整本教材的智能问答系统。实现文档切分与向量化、知识库构建、语义检索、基于检索结果的生成、答案引用来源标注和题目推荐六个进阶功能模块，支持章节知识问答、跨章节比较、复习提纲生成和知识点自测推荐四类查询。

## 2 相关技术简介

**大语言模型**：基于Transformer架构的预训练语言模型，通过在海量文本数据上进行自监督学习，具备了通用的语言理解和生成能力。本系统使用的通义千问（qwen-plus）是阿里云发布的中文大语言模型，在多项中文NLP基准测试中表现优异。

**Streamlit**：一个面向数据科学的Python Web框架，允许开发者仅通过Python代码构建交互式Web应用，无需编写HTML、CSS和JavaScript。本研究使用Streamlit搭建系统的前端交互界面。

**Prompt工程**：通过设计提示文本引导大语言模型按预期方式输出的技术。提示的设计直接影响模型输出的质量、格式和稳定性。本研究中两种方案的差异本质上源于不同的提示设计策略。

**OpenAI兼容API**：OpenAI定义的RESTful API规范已成为行业事实标准。阿里云百炼平台提供兼容该规范的API接口，开发者可使用OpenAI Python SDK直接调用通义千问模型。

**PyPDF2 / python-docx**：PyPDF2是Python的PDF文件处理库，可从PDF页面中提取文本内容。python-docx用于读取和处理Microsoft Word（.docx）格式的文档。

**Embedding（文本向量化）**：将文本转换为固定维度的数值向量的技术。语义相近的文本在向量空间中距离更近。本系统使用阿里云text-embedding-v2模型将文档块和用户问题转化为1536维向量，为语义检索提供数学基础。

**RAG（检索增强生成）**：Retrieval-Augmented Generation，2020年由Meta AI提出。核心思想是在大模型生成回答前，先从外部知识库中检索相关文档作为上下文，再将检索结果与问题一同送入模型，使模型能够基于提供的资料作答而非仅凭记忆，有效减少幻觉问题并支持答案溯源。

**余弦相似度**：衡量两个向量方向相似程度的数学方法。取值范围[-1, 1]，值越接近1表示语义越相近。本系统用余弦相似度作为检索阶段的核心度量，计算问题向量与知识库中每个文档块向量的相似度，取最相关的top-k个块作为生成上下文。

## 3 算法模型

### 3.1 系统整体架构

本系统采用前后端分离的架构设计。前端基于Streamlit框架实现Web交互界面；后端通过阿里云百炼API调用通义千问大语言模型。系统架构如图：

```
用户 → Web界面(Streamlit) → 文件上传/功能操作
                                  ↓
                           功能模块调用
                （summarizer / outline / quiz / exporter）
                                  ↓
                       API调用 (OpenAI SDK)
                                  ↓
                   通义千问模型 (qwen-plus)
                                  ↓
                         返回结果 → 界面展示
```

### 3.2 摘要生成方案设计

**方案一：单轮直接生成**

将课程文本与格式要求拼接为单条提示，一次性发送给模型，要求模型直接输出200-300字的课程摘要。单次API调用完成。

提示设计思路：在prompt中明确字数范围（200-300字）、内容要求（概括核心概念和关键知识点）、风格要求（简洁专业，不添加个人观点）三重约束。

**方案二：多轮分步迭代**

将摘要生成过程拆解为三个子步骤：

（1）**知识点提取**：要求模型从课程资料中提取5-8个核心知识点，每个知识点用一句话概括。这一步让模型聚焦于"识别重点"的单一任务。

（2）**摘要生成**：将第一步提取的知识点作为输入，要求模型基于这些知识点生成一篇连贯的摘要。这一步输入是精炼后的知识点而非原始长文，降低了信息筛选的难度。

（3）**字数压缩**：将第二步生成的摘要输入模型，要求将其压缩到200-300字之间，保持核心信息完整。这一步专门解决字数控制问题。

### 3.3 选择题生成方案设计

**方案一：单轮直接生成**

将课程文本与选择题格式要求拼接为单条提示，要求模型直接输出包含5道四选一选择题的JSON数组，每题包含question、options、answer、explanation四个字段。

**方案二：多轮分步迭代**

将选择题生成过程拆解为三个子步骤：

（1）**知识点提取**：要求模型从课程资料中提取5个最适合出选择题的知识点，确保题目覆盖不同知识域。

（2）**逐知识点出题**：将知识点列表作为输入，要求模型为每个知识点生成一道四选一选择题，输出JSON格式。

（3）**质量校验**：将第二步生成的题目提交给模型进行自我检查，修正标准包括：每题有唯一正确答案、错误选项表面合理但实质错误、解析须说明对错原因、题目之间无知识点重叠。

### 3.4 提纲生成

提纲生成采用单次推理方案。提示中要求模型使用Markdown格式组织分层知识点（## 主层级、### 子层级、- 具体要点），每个知识点用一句话概括，知识点间有逻辑递进关系。

### 3.5 模型参数设置

| 参数 | 摘要任务 | 选择题任务 |
|------|---------|-----------|
| model | qwen-plus | qwen-plus |
| temperature | 0.3 | 0.3/0.5 |
| top_p | 默认(1.0) | 默认(1.0) |

摘要和提纲任务使用较低的temperature（0.3）以保持输出的确定性和准确性。选择题的出题阶段使用稍高的temperature（0.5）以增加题目多样性，校验阶段使用0.3以保证质量评估的严谨性。

### 3.6 进阶功能RAG架构设计（进阶功能模块1-3）

进阶功能部分在项目基础上引入RAG架构，系统架构升级为：

```
用户上传PDF → 文本提取 → 文档切分(chunker.py)
                              ↓
                    每块调Embedding API
                              ↓
                    向量存入知识库(JSON)
                              ↓
用户提问 → 问题向量化 → 余弦相似度检索 → 取top-k相关块
                              ↓
                  prompt = 检索内容 + 问题 + 格式要求
                              ↓
                       通义千问生成回答
                              ↓
                    回答 + 【引用来源】标注
```

**文档切分策略**：采用二级策略。优先按段落边界切分，保持语义完整性；若段落式切分得到的最大块超过chunk_size的1.5倍（750字），则退化为固定窗口滑动切分（500字/块，重叠100字），确保任何情况下系统不会因超长段落而失效。

**向量化与知识库**：使用阿里云text-embedding-v2模型将每个文本块转化为1536维浮点向量。所有(文本块, 向量)对存入本地JSON文件实现持久化，系统重启后无需重新向量化。向量和文本块以平行列表存储，索引天然对应。

**语义检索**：采用余弦相似度度量。用户问题同样经过Embedding API向量化，与知识库中每个文档块的向量计算余弦值，按得分降序排列取top-k。章节问答取3块，跨章节比较和复习提纲取5块以扩大覆盖范围。

### 3.7 基于检索结果的生成（进阶功能模块4-5）

检索完成后，将检索到的文本块以分隔符拼接，与用户问题及回答要求一同构造prompt，发送给qwen-plus模型。模型被限定为"基于以下课程资料回答"，其输出质量直接取决于检索质量——检索到的文档块越相关，回答越准确。每个回答末尾强制标注【引用来源】，说明答案依据的资料内容，实现可追溯性。

### 3.8 四类查询设计（进阶功能模块7）

系统支持以下四种查询类型，各类通过不同的prompt设计引导模型输出：

**章节知识问答**：检索top-3块，prompt要求基于资料直接回答，引用具体内容，资料覆盖不足时明确说明。

**跨章节比较分析**：检索top-5块以扩大章节覆盖，prompt要求分别列举各章节相关知识点，以表格或分点形式对比异同，标注资料出处。

**复习提纲生成**：检索top-5块，prompt要求Markdown层级格式（##/###/ -），用⭐标注重要程度，末尾列出5个常见易混淆知识点并做辨析。

**知识点自测推荐**：检索top-3块，prompt要求模型扮演诊断角色——先识别3个学生可能薄弱的环节，再为每个薄弱点出一道四选一选择题，最后给出针对性学习计划，输出JSON格式。

### 3.9 题目推荐机制（进阶功能模块6）

题目推荐模块（exercise_recommender.py）采用"检索匹配+自动生成兜底"的混合策略。第一步从本地题库JSON文件中通过关键词匹配检索与用户指定主题相关的题目；若匹配不足3道，则调用LLM基于知识库检索结果即时生成补充题目。题库通过项目部分的选择题生成功能持续积累，使用次数越多，匹配命中率越高，实现"越用越智能"的冷启动友好设计。

## 4 实验分析

### 4.1 数据处理

**实验文本**：选取自然语言处理课程相关文本作为测试样本，涵盖NLP基础概念、核心任务体系、Transformer架构、BERT与GPT预训练模型、大语言模型等章节内容。文本长度约500字。

**测试环境**：
- 操作系统：Windows 11
- 编程语言：Python 3.13
- 模型服务：阿里云百炼API（通义千问qwen-plus）
- API配额：标准按量计费

**评估指标**：
- 客观指标：输出字数、API调用次数、推理耗时
- 主观指标：准确性（10分）、可读性（10分）、字数符合度（10分）/ 题目覆盖度（10分）、解析质量（10分）、格式规范性（10分）

### 4.2 模型设置

本实验采用相同模型、相同输入文本、相同temperature参数的设置，唯一变量为推理方案的不同——方案一为单轮直接生成，方案二为多轮分步迭代。这确保了实验对照的单一变量原则。

### 4.3 实验结果

**摘要生成对比**

| 指标 | 方案一（单轮） | 方案二（多轮） |
|------|--------------|--------------|
| 输出字数 | 340字 | 300字 |
| 推理耗时 | 5.0秒 | 10.8秒 |
| API调用次数 | 1次 | 3次 |

**选择题生成对比**

| 指标 | 方案一（单轮） | 方案二（多轮） |
|------|--------------|--------------|
| 题目数量 | 5道 | 5道 |
| 推理耗时 | 18.7秒 | 56.0秒 |
| API调用次数 | 1次 | 3次 |
| JSON格式合规率 | 约90% | 约98% |

**综合对比**

| 指标 | 方案一（单轮） | 方案二（多轮） |
|------|--------------|--------------|
| 摘要+选择题总耗时 | 23.8秒 | 66.8秒 |
| 总API调用次数 | 2次 | 6次 |

### 4.4 对比分析

**输出质量分析**

方案二（多轮分步迭代）在字数控制方面表现更优。实验数据显示，方案一输出340字，超出要求的200-300字范围；方案二经过独立压缩步骤后输出300字，精准落入目标区间。这验证了任务拆解策略对输出可控性的提升。

在选择题方面，方案二的第三步（质量校验）通过让模型重新审视已生成的题目，有效修正了部分选项表述不严谨、解析不完整等问题。方案一的JSON输出偶有格式问题（如被Markdown代码块包裹），需额外清理；方案二经过校验步骤后，格式规范性更高。

**计算成本分析**

方案二的API调用次数是方案一的3倍，总耗时约为2.8倍（23.8秒 vs 66.8秒）。在相同文本输入下，方案一总API调用2次，方案二总API调用6次。按通义千问qwen-plus的计费标准，单次测试的额外成本极低，在实际应用中可以忽略。

**策略适用场景分析**

方案一（单轮直接生成）适用于：
- 对耗时敏感的场景（如实时交互）
- 简单明确的摘要任务
- 输出格式要求不严格的情况

方案二（多轮分步迭代）适用于：
- 对输出质量要求高的场景（如正式课程资料发布）
- 输出格式有严格规范的任务（如标准化试题出题）
- 输入文本较长、信息密度较高的情况
- 需要稳定字数控制的场景

### 4.5 进阶功能RAG功能验证

为验证进阶功能RAG模块的实际效果，采用一段涵盖四个章节（NLP概述、Transformer架构、预训练语言模型、大语言模型）的课程文本作为知识库输入，测试四类查询功能。

**知识库构建**

| 指标 | 数值 |
|------|------|
| 原始文本长度 | 约1000字 |
| 切分块数 | 2块 |
| 向量维度 | 1536维 |
| 构建耗时 | 约3-5秒 |

**四类查询测试结果**

| 查询类型 | 测试问题 | 检索块数 | 回答质量 |
|---------|---------|---------|---------|
| 章节问答 | "BERT和GPT有什么区别？" | top-3 | 准确列出架构、预训练任务、擅长方向三方面差异，末尾标注引用来源 |
| 跨章节比较 | "对比第一章NLP基础任务和第三章预训练模型" | top-5 | 以表格形式呈现两章异同，分析NLP基础任务到预训练模型的技术演进关系 |
| 复习提纲 | "Transformer架构" | top-5 | 输出⭐标注的Markdown层级提纲，含5个易混淆知识点辨析 |
| 自测推荐 | "注意力机制" | top-3 | 识别3个薄弱环节，各出一道选择题并给出学习计划，JSON格式规范 |

**习题推荐测试**

通过项目选择题功能生成5道题目自动存入题库。以"注意力机制"为主题进行习题推荐，题库关键词匹配命中2题，自动生成3题补充，共返回5道推荐题目。

## 5 编码与实现

### 5.1 项目结构

```

├── config.py               # API连接配置
├── file_importer.py        # 资料导入
├── summarizer.py           # 摘要生成（双方案）
├── outline_generator.py    # 知识点提纲生成
├── quiz_generator.py       # 选择题生成（双方案）
├── exporter.py             # 结果导出
├── experiment.py           # 命令行对比实验脚本
├── chunker.py              # 文档切分与向量化
├── vector_store.py         # (3)：向量化+知识库+检索
├── qa_generator.py         # (5)：RAG生成+引用来源
├── exercise_recommender.py # 题目推荐模块
├── app.py                  # Streamlit Web主界面（项目+进阶功能）
├── 启动.bat                # 一键启动脚本
└── 代码逐行解释文档.md      # 代码文档
```

### 5.2 核心实现要点

**—— 资料导入**：通过file_importer.py实现统一的文件读取接口。利用PyPDF2的PdfReader类解析PDF文档，逐页提取文本并拼接；利用python-docx的Document类解析Word文档，遍历段落提取文本。通过文件名后缀判断格式，返回统一编码的UTF-8字符串。

**—— 摘要生成**：summarizer.py中定义summary_single()和summary_multi()两个函数。方案一构造单条prompt发送API请求；方案二通过三次API调用依次实现知识点提取、摘要生成、字数压缩，每次调用以上一步的输出作为输入。compare_summary()函数同时运行两个方案并返回结果元组。

**—— 提纲生成**：outline_generator.py通过指定Markdown层级格式的prompt引导模型输出结构化提纲。模型返回的Markdown内容可直接在Streamlit界面渲染。

**—— 选择题生成**：quiz_generator.py是技术实现最复杂的模块。方案一要求模型输出纯JSON数组，通过replace()方法清除可能的Markdown代码块标记，使用json.loads()解析并辅以try-except容错。方案二在方案一基础上增加知识点提取和校验修正两个子步骤，校验步骤中通过json.dumps()将待检查题目传回模型。

**—— 结果导出**：exporter.py实现三种格式导出。JSON导出使用json.dump()配合ensure_ascii=False和indent=2参数保证中文可读；Markdown导出使用f-string拼接模板；HTML导出内嵌CSS样式实现专业排版。

**Web界面**：app.py基于Streamlit框架，使用st.sidebar放置文件上传组件，st.tabs组织功能模块，st.columns(2)实现左右分栏对比展示，st.session_state管理跨交互状态。方案一用st.info（蓝色框）展示，方案二用st.success（绿色框）展示，视觉上清晰区分。进阶功能RAG问答模块通过st.radio水平按钮切换四种查询类型，构建知识库按钮嵌入侧边栏实现一键构建。

**—— 文档切分**：chunker.py实现二级切分策略。优先用正则`\n\s*\n`按段落边界切分，保证语义完整；若段落切分失败（无段落边界或单块过长），退化为500字固定窗口滑动切分（重叠100字），通过`end >= len(text)`终止条件防止死循环。

**(3) —— 向量化与检索**：vector_store.py中VectorStore类封装全部知识库操作。get_embedding()调用阿里云text-embedding-v2 API；add()实现批量入库；search()实现余弦相似度检索，通过`sum(a*b)`点积与`**0.5`开方运算计算，分母加`1e-8`防除零；_save()和_load()使用json.dump/json.load实现知识库本地持久化，ensure_ascii=False保证中文正常存储。

**(5) —— RAG生成**：qa_generator.py中四个查询函数遵循统一的"检索→拼context→构造prompt→调LLM→返回"流程。prompt设计通过角色设定（"你是一个课程助教"）、行为约束（"基于以下课程资料回答"）、格式要求（Markdown/JSON/引用来源）三层结构引导模型输出。init_knowledge_base()串联chunker和vector_store，作为一键构建入口。

**—— 题目推荐**：exercise_recommender.py采用"关键词匹配+自动生成兜底"策略。save_quiz_bank()通过json.load读已有题库→extend追加→json.dump写回，实现题库累积。recommend_by_topic()先对所有题目做关键词匹配，命中不足3题时调LLM基于知识库检索结果生成补充题目，最多返回5题。

### 5.3 关键技术难点

**JSON解析容错**：模型偶尔在JSON外围添加Markdown代码块标记或解释性文本。解决方案是在json.loads()前用replace()清理，并加入try-except异常处理，确保任何格式偏差都不会导致系统崩溃。

**字数精准控制**：单轮方案中模型对字数约束的遵循不够稳定。解决方案是在多轮方案中引入独立的压缩步骤，让模型专门处理字数控制这一单一任务，效果显著提升。

**多轮上下文衔接**：多轮迭代中需要保证每步之间的信息传递完整。解决方案是每步的prompt中显式引用上一步的输出变量（通过f-string嵌入），并添加明确的衔接指令。

**进阶功能难点一：文本切分边界处理**：直接按500字窗口切分导致句子被拦腰截断，如"掩码语言模型"在一块而"（MLM）进行预训练"在另一块，检索时语义不完整。解决方案是采用双层策略——优先按段落边界切分保语义，超长段落退化为滑动窗口切分防崩溃。

**进阶功能难点二：Embedding模型调用稳定性**：text-embedding-v2 API在批量调用时偶有超时。虽然目前每块逐条调用确保了稳定性，但未来优化方向是支持批量化调用以提升多块处理效率。

**进阶功能难点三：知识库持久化与一致性**：向量化和文本块存入JSON后，若后续新增文档导致JSON文件越来越大，加载耗时线性增长。当前采用全量读写方案适配课程资料规模，更大规模场景需升级为向量数据库（如ChromaDB/Milvus）实现增量写入和索引加速。

## 6 总结与展望

### 6.1 工作总结

本研究完成了课程资料智能摘要与测验生成系统的设计与实现，系统包含资料导入、摘要生成、知识点提纲生成、选择题生成和结果导出五个功能模块，基于Streamlit框架搭建了Web交互界面。技术上采用阿里云百炼的通义千问大语言模型API，通过OpenAI兼容接口实现模型调用。

本研究设计了单轮直接生成与多轮分步迭代两种推理方案的对比实验。实验结果表明：（1）多轮分步迭代在字数控制、格式规范性方面优于单轮直接生成，核心原因在于将复杂任务拆解为单一职责的子任务，降低了每步的推理难度；（2）单轮方案在速度和计算成本方面具有明显优势，适用于对实时性要求较高的场景；（3）多轮方案的第三步（校验/压缩）对最终输出质量起到关键的兜底作用。

在进阶功能部分，本研究成功将系统从单章处理扩展为支持整本教材的RAG智能问答系统。实现了文档切分与向量化、知识库构建、语义检索、基于检索结果的生成、答案引用来源标注和题目推荐六个进阶功能模块。系统支持四类查询：章节知识问答、跨章节比较分析、复习提纲生成和知识点自测推荐。知识库采用本地JSON持久化方案，检索层和生成层解耦设计确保了模型的可替换性。RAG架构有效解决了大语言模型的幻觉问题和知识截止日期限制，回答基于检索到的真实课程资料生成，具备可追溯性。

### 6.2 未来展望

一是引入向量数据库和检索增强生成技术，使系统能够处理整本教材级别的长文档，支持跨章节知识问答和比较分析。二是设计少样本示例提示策略，通过在prompt中加入高质量的摘要/选择题示例，探索在不增加API调用次数的情况下提升输出质量的可能性。三是支持更多模型接入和切换，实现不同规模模型的横向对比分析。
